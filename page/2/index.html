开始学习 Python 之前的几个疑问

Why Do People Use Python?

  • Software quality
    The uniformity of Python code makes it easy to understand.

  • Developer productivity
    Python code is typically one-third to one-fifth the size of equivalent C++ or Java code.That means there is less to type, less to debug, and less to maintain.

  • Program portability
    Most Python programs run unchanged on all major computer platforms.

  • Support libraries

  • Component integration
    Python code can invoke C and C++ libraries, can be called from C and C++ programs, can integrate with Java and .NET components.

  • Enjoyment
    Because of Python’s ease of use and built-in toolset, it can make the act of programming more pleasure than chore.

Is Python a “Scripting Language”?

general-purpose programming language that blends procedural, functional, and object-oriented paradigms

  • Shell tools
    coding operating-system-oriented scripts

  • Control language
    control and direct other application components.

What’s the Downside?

Execution speed is the only major downside to Python.

What Can I Do with Python?

  • Systems Programming
  • GUIs
  • Internet Scripting
  • Component Integration
  • Database Programming
  • Rapid Prototyping
  • Numeric and Scientific Programming
  • And More: Gaming, Images, Data Mining, Robots, Excel…

How Is Python Developed and Supported?

  • The PSF (Python Software Foundation)
  • Open Source Tradeoffs

What Are Python’s Technical Strengths?

  • Object-Oriented and Functional
  • Free
  • Portable
  • Powerful
  • Mixable
  • Relatively Easy to Use
  • Relatively Easy to Learn

How Python Runs Programs?

  • source (m.py) –> Byte code (m.pyc) –> Runtime (PVM)

The java.util.function Package

The java.util.function package was added in Java 8.
The interfaces in java.util.function fall into four categories.

  • consumers : take a generic argument and return nothing
  • suppliers : take no arguments and return a value
  • predicates : take an argument and return a boolean
  • functions : take a single argument and return a value

[数学文章作法 推敲篇] 读书笔记

读完结城浩老师的 [数学文章作法 推敲篇] 一书，受益匪浅。
此书的内容其实不仅仅限于数学文章，
不论是日常写作还是商业邮件，都能从中得到帮助。
以下是根据书中检查表翻译后的笔记。

读者的迷惑

  • 读者通过文章构筑概念。构筑概念就像搭积木。是否给了读者正确的积木？
  • 是否给读者展示了没有分支的一条道？
  • 语句的表记是否统一？
  • 是否存在过长的句子？
  • 是否存在语言的不足？
  • 是否存在应该用图表传达的概念，却用了文章？
  • 是否存在多余的语言？
  • 指示语的所指是否明确？

推敲的基本

  • 重读的时候是否戴着读者的帽子？
  • 重读的时候手里是否拿着笔记用具？
  • 是否找到与想写的事情的差异？
  • 是否持有文章完成形象？
  • 是否删除不要的东西？
  • 是否补充必要的东西？
  • 是否将读者卷入迷雾中？
  • 是否反复读了文章？
  • 是否意识着观点再读？
  • 是否放置一段时间然后再读？
  • 是否持有作为作者的自觉性和责任感？

词语

  • 是否使用读者能够理解的词语？
  • 是否检讨过更合适的词语？
  • 是否将专业术语和一般词语区别开来？
  • 是否正确的使用专业术语？
  • 是否在有必要的情况下，记述词语的定义？
  • 是否注意定义的词性？
  • 是否欠缺考虑的创造新词？
  • 是否存在无意中使用的词语？比如说”基本上来说”，”在某种意义上”，”真正”，”绝对”，”一定”等。
  • 指示语所指示的对象是否确定？

句子的推敲

  • 是否存在过长的句子？
  • 是否存在容易被误解的复句？
  • 符号的位置和词序是否恰当？
  • 是否存在不要的句子？
  • 是否存在使用多个”的”字的句子？
  • 当存在多个概念时，对应关系是否明确？
  • 是否正确使用”各自”？
  • 是否存在容易被误解的”言外之意”？
  • 是否写有”A的情况”和”A以外的情况”？
  • 是否将复杂的概念强行写成文章？
  • 是否明确将什么与什么作比较？
  • 是否明确比较的基准？
  • 是否避免双重否定？

文章整体的平衡

  • 文章整体上，是否存在欠缺的要素？
  • 是否能一丝不漏的把读者的期望写出来？
  • 各要素的长度和强调的程度是否合适？
  • 文章是否能抵抗误读？
  • 为了避免误读，是否加入了适当的冗余性？
  • 是否存在粗糙的部分混在其中？
  • 是否整体的读过？（车轮战）
  • 是否改变观点读过好多次？（阶段战）
  • 是否仔细阅读修改较多的地方？（张弛战）

审查

  • 是否考虑让他人审查？
  • 是否考虑审查者是谁？（对象读者，专家，对文章敏感的人等）
  • 是否给了审查者审查项目？
  • 是否传达了审查观点？
  • 是否考虑使用审查专用网页？
  • 是否认真确认回馈信息？
  • 是否将回馈信息分类，整理？
  • 是否冷静的接受回馈信息？
  • 是否以作者的责任感来对应回馈信息？
  • 是否把握没有对应的回馈信息？
  • 是否和审查者保持着信赖关系？

推敲的窍门

  • 是否尽力保守期限？
  • 是否把握到期限为止的总时间？
  • 是否把握有无整段时间可用？
  • 为了有效的推敲，是否制作词汇表？
  • 是否使用检索，替换，和校准功能？
  • 是否掌握自己写文章时的习惯？
  • 是否对文件进行备份？
  • 是否使用版本管理系统？
  • 是否使用多种读法阅读？（出声读，画面读，纸面读，换地方读，随机的读某页等）

靠巧合编程 (Programming by Coincidence)

老式的黑白战争片：一个士兵站在一片空旷地，那里有地雷吗？他试探性的用枪挑了挑，发现没事于是大步走着被炸成了碎片。

士兵起初的探测没有发现地雷，但这只不过是侥幸，他由此得出了错误的结论。

怎样靠巧合编程 (How to Program by Coincidence)

Fred不知道代码为什么会失败，因为他一开始就不知道它为什么会工作。

即使是修改的代码可以工作，那也很可能仅仅是一个巧合。

  • 语境的偶然 (Accidents of Context)
    假设你正在编写一个 utility 模块， 只是因为你现在实在为GUI环境编写代码，该模块就必须依存GUI吗？你还依存什么没有保障的东西？

  • 隐含的假定 (Implicit Assumptions)
    巧合可以在所有层面上让人误入歧途。在所有层面上人们带着许多的假定工作，但是这些假定很少被写入文档，甚至在不同开发者之间是相互冲突的。
    并非以明确的事实为基础的假定是所有项目的祸害。
    不要假定它，证明它。don’t assume it, prove it.

怎样深思熟虑的编程

  • 总是意识到你在做什么。
  • 不要盲目的编程。试图构建不完全理解的应用，或是使用不熟悉的技术，就是希望自己被巧合误导。
  • 按照计划行事。
  • 依靠可靠的事物。
  • 为你的假定建立文档。
  • 不要只是测试你的代码，还要测试你的假定。
  • 为你的工作划分优先级，把时间花在重要的方面。
  • 不要做历史的奴隶。不要让已有的代码支配将来的代码。
