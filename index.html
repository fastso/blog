Default Methods in Interfaces

Provide an implementation of a method inside an interface.

An Employee interface with a default method

Static Methods in Interfaces

Provide a class-level utility method to an interface, along with an implementation.
Static methods in interfaces remove the need to create separate utility classes,
though that option is still available if a design calls for it.

key points

  • Static methods must have an implementation
  • Cannot override a static method
  • Call static methods from the interface name
  • Do not need to implement an interface to use its static methods

[数学文章作法 推敲篇] 读书笔记

读完结城浩老师的 [数学文章作法 推敲篇] 一书，受益匪浅。
此书的内容其实不仅仅限于数学文章，
不论是日常写作还是商业邮件，都能从中得到帮助。
以下是根据书中检查表翻译后的笔记。

读者的迷惑

  • 读者通过文章构筑概念。构筑概念就像搭积木。是否给了读者正确的积木？
  • 是否给读者展示了没有分支的一条道？
  • 语句的表记是否统一？
  • 是否存在过长的句子？
  • 是否存在语言的不足？
  • 是否存在应该用图表传达的概念，却用了文章？
  • 是否存在多余的语言？
  • 指示语的所指是否明确？

推敲的基本

  • 重读的时候是否戴着读者的帽子？
  • 重读的时候手里是否拿着笔记用具？
  • 是否找到与想写的事情的差异？
  • 是否持有文章完成形象？
  • 是否删除不要的东西？
  • 是否补充必要的东西？
  • 是否将读者卷入迷雾中？
  • 是否反复读了文章？
  • 是否意识着观点再读？
  • 是否放置一段时间然后再读？
  • 是否持有作为作者的自觉性和责任感？

词语

  • 是否使用读者能够理解的词语？
  • 是否检讨过更合适的词语？
  • 是否将专业术语和一般词语区别开来？
  • 是否正确的使用专业术语？
  • 是否在有必要的情况下，记述词语的定义？
  • 是否注意定义的词性？
  • 是否欠缺考虑的创造新词？
  • 是否存在无意中使用的词语？比如说”基本上来说”，”在某种意义上”，”真正”，”绝对”，”一定”等。
  • 指示语所指示的对象是否确定？

句子的推敲

  • 是否存在过长的句子？
  • 是否存在容易被误解的复句？
  • 符号的位置和词序是否恰当？
  • 是否存在不要的句子？
  • 是否存在使用多个”的”字的句子？
  • 当存在多个概念时，对应关系是否明确？
  • 是否正确使用”各自”？
  • 是否存在容易被误解的”言外之意”？
  • 是否写有”A的情况”和”A以外的情况”？
  • 是否将复杂的概念强行写成文章？
  • 是否明确将什么与什么作比较？
  • 是否明确比较的基准？
  • 是否避免双重否定？

文章整体的平衡

  • 文章整体上，是否存在欠缺的要素？
  • 是否能一丝不漏的把读者的期望写出来？
  • 各要素的长度和强调的程度是否合适？
  • 文章是否能抵抗误读？
  • 为了避免误读，是否加入了适当的冗余性？
  • 是否存在粗糙的部分混在其中？
  • 是否整体的读过？（车轮战）
  • 是否改变观点读过好多次？（阶段战）
  • 是否仔细阅读修改较多的地方？（张弛战）

审查

  • 是否考虑让他人审查？
  • 是否考虑审查者是谁？（对象读者，专家，对文章敏感的人等）
  • 是否给了审查者审查项目？
  • 是否传达了审查观点？
  • 是否考虑使用审查专用网页？
  • 是否认真确认回馈信息？
  • 是否将回馈信息分类，整理？
  • 是否冷静的接受回馈信息？
  • 是否以作者的责任感来对应回馈信息？
  • 是否把握没有对应的回馈信息？
  • 是否和审查者保持着信赖关系？

推敲的窍门

  • 是否尽力保守期限？
  • 是否把握到期限为止的总时间？
  • 是否把握有无整段时间可用？
  • 为了有效的推敲，是否制作词汇表？
  • 是否使用检索，替换，和校准功能？
  • 是否掌握自己写文章时的习惯？
  • 是否对文件进行备份？
  • 是否使用版本管理系统？
  • 是否使用多种读法阅读？（出声读，画面读，纸面读，换地方读，随机的读某页等）

靠巧合编程 (Programming by Coincidence)

老式的黑白战争片：一个士兵站在一片空旷地，那里有地雷吗？他试探性的用枪挑了挑，发现没事于是大步走着被炸成了碎片。

士兵起初的探测没有发现地雷，但这只不过是侥幸，他由此得出了错误的结论。

怎样靠巧合编程 (How to Program by Coincidence)

Fred不知道代码为什么会失败，因为他一开始就不知道它为什么会工作。

即使是修改的代码可以工作，那也很可能仅仅是一个巧合。

  • 语境的偶然 (Accidents of Context)
    假设你正在编写一个 utility 模块， 只是因为你现在实在为GUI环境编写代码，该模块就必须依存GUI吗？你还依存什么没有保障的东西？

  • 隐含的假定 (Implicit Assumptions)
    巧合可以在所有层面上让人误入歧途。在所有层面上人们带着许多的假定工作，但是这些假定很少被写入文档，甚至在不同开发者之间是相互冲突的。
    并非以明确的事实为基础的假定是所有项目的祸害。
    不要假定它，证明它。don’t assume it, prove it.

怎样深思熟虑的编程

  • 总是意识到你在做什么。
  • 不要盲目的编程。试图构建不完全理解的应用，或是使用不熟悉的技术，就是希望自己被巧合误导。
  • 按照计划行事。
  • 依靠可靠的事物。
  • 为你的假定建立文档。
  • 不要只是测试你的代码，还要测试你的假定。
  • 为你的工作划分优先级，把时间花在重要的方面。
  • 不要做历史的奴隶。不要让已有的代码支配将来的代码。

黑板 (Blackboards)

黑板方法的一些关键特性：

  • 没有侦探知道其他任何侦探的存在–他们查看黑板，从中了解新的信息，并且加上他们的发现。
  • 侦探可能接受过不同的训练，有不同程度的教育背景和专业经验。他们的唯一共同点就是都渴望破案。
  • 在这个过程中，不同的侦探会来来去去，工作班次也可能不同。
  • 放在黑板上的内容没有限制，可以是图片，判断，物证等等。

黑板实现 (Blackboard Implementations)

现代的分布式类黑板结构 (distributed blackboard-like systems) 有 JavaSpaces 和 T Spaces。
JavaSpaces的主要操作：

Name Function
read 在空间中查找并读取数据
write 把数据写入空间
take 与read类似，但同时从空间中移除数据
notify 设定每当写入与模板匹配的对象时就发出通知

它只是视图 (It's just a view)

那人依然只听到
他想要听到的东西
而不顾其他

Still, a man hears
What he wants to hear
And disregards the rest
–Simon and Garfunkel, “The Boxer”

我们应该把程序性划分成模块，每个模块都有其自身的责任。
这会产生一些新的问题，模块之间怎样相互交谈？怎样管理他们的逻辑依赖？如何对不同模块的状态变化进行同步？
我们想要每个模块只听到它想听到的东西。

发布/订阅 (Publish/Subscribe)

对象应该能进行登记，只接受他们需要的事件。

Model-View-Controller

关键概念：既让模型与表示模型的GUI分离，也让模型与管理视图的控件分离。
这样做，可以支持同一数据模型的多个视图，可以在许多不同的数据模型上使用公共的查看器，甚至还可以使用多个控制器提供非传统的输入机制。

超越GUI (Beyond GUIs)

MVC不只是在GUI开发中，它是一种通用的编程技术。
视图是对模型的一种解释，它无需是图形化的。控制器更是一种协调机制，不一定要与任何种类的输入设备有关。

  • 模型：目标对象的抽象数据模型。模型对任何视图和控制器都没有直接的了解。
  • 视图：解释模型的方式。它订阅模型中的变化和来自控制器的逻辑事件。
  • 控制器：控制视图。并向模型提供新数据的途径。它既向模型，也向视图发布事件。

仍然有耦合 (Still Coupled)

尽管耦合减少了，但订阅者和发布者间仍然需要有一些相互了解。
比如Java中必须对公共接口定义和调用约定达成一致。

时间耦合 (Temporal Coupling)

时间是软件架构中常常被忽视。
时间有两个方面很重要:

  • 并发(事情在同一时间发生)
  • 次序(事情在时间中的相对位置)

如果在编程时，不对时间进行思考，会带来时间耦合。时间耦合的例子如下：

  • 方法 A 总是在方法 B 之前调用
  • 同时只能运行一个报告
  • 在接收到按钮点击前，你必须等待屏幕重画
  • 嘀必须发生在嗒之前

我们有必要在工作流分析，架构，设计，部署等开发领域中减少任何基于时间的依赖。

工作流 (Workflow)

架构 (Architecture)

饥饿的消费者模型 (the hungry consumer model)
用多个独立的消费者任务 (independent consumer tasks) 和一个集中式工作队列 centralized work queue() 取代中央调度器 (central scheduler)。
各个消费者任务从工作队列中抓取一项，并对其进行处理。当各个任务完成其工作时，回到工作队列抓取下一项。
这样，如果任何任务陷入停滞，其他任务可以利用这一空闲，各个组件都可以按照自己的步伐前进，每个组件都在时间上解除了与其他组件的耦合。

实际上我们不是在创建组件，而是服务 (定义良好，一致接口的独立，并发对象)。

并发设计 (Design for Concurrency)

必须对全局或静态变量加以保护，使其免于并发访问。
不管调用的次序是什么，都要确保给出的是一致的状态信息。状态信息如果在某些调用之间处于无效状态，你就是在依赖一个巧合：没有人会在那个时间调用你的对象。

对并发和次序依赖进行思考还能够引导你设计更整洁的接口 (Cleaner Interfaces)。

部署 (Deployment)

一旦设计了具有并发要素的架构，就可以灵活的处理应用的部署方式：单机，客户端-服务器，或者n层 (n-tier)。

元程序设计 (Metaprogramming)

没有多少天才可以克服对细节的关注。 –Levy第八法则

No amount of genius can overcome a preoccupation with detail. –Levy’s Eighth Law

动态配置 (Dynamic Configuration)

要用元数据描述应用的配置选项：调谐参数，用户偏好，安装目录等等。
Use metadata to describe configuration options for an application: tuning parameters, user preferences, the installation directory, and so on.

Metadata是什么？
严格的说，Metadata是关于数据的数据。
从宽泛的意义上来讲，Metadata是任何对应用进行描述的数据–应用该怎样运行，它应该使用什么资源等等。典型的情况下，Metadata在运行时，而不是编译时被访问和使用。

元数据驱动的应用 (Metadata-Driven Applications)

元数据驱动的好处：

  • 迫使解耦，带来更灵活，可适应性更好的程序。
  • 迫使推迟细节处理，创建更强健，更抽象的设计。
  • 无需重新编译应用。
  • 可以通过接近问题领域 (problem domain) 的方式表示元数据。
  • 可以使用相同的应用引擎和不同的元数据，实现不同的项目。

解耦与得墨忒耳法则 (Decoupling and the law of demeter)

使耦合减至最少 (Minimize Coupling)

函数的得墨忒耳法则 (The Law of Demeter for Functions)

函数的得墨忒耳法则规定，某个对象的任何方法都应该只调用属于以下情形的方法：

  • 函数自身
  • 传入该方法的任何参数
  • 它创建的任何对象
  • 任何直接持有的组件
